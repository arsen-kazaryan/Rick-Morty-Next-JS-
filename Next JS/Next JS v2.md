# Next Js 

## Установка и создание проекта 

- Установка 
```bash 
  npx create-next-app@latest
```
### Рекомендумые настройки при init
* TS: `Yes`
* ESlint: `Yes` (Поиск ошибок в коде)
* Tailwind CSS : `Yes` (утилитарный CSS-фреймворк для быстрой верстки прямо в классах элементов)
* Src directory: `Yes`
* App Router: `Yes` (современный стандарт маршрутизации Next.js)
* Turbopack: `Yes` (Быстрый сборщик)

## Layout(Макеты )
`Layout`- это общая оболочка (каркас) для страниц приложения(один layout - он обязательный).  
*Например*
```
Header

Меняем страницы

Footer
```
- Header и Footer останутся. Изменится только содержимое страницы.

### Вложенные Layout 
```
app
│
├── layout.tsx
│
├── dashboard
│      layout.tsx
│
│      page.tsx
│
└── profile
       page.tsx
```
*При открытии dasboard используются сразу 2 layout*  

**Когда использовать**
-  общая шапка/футер для всего сайта
-  своя навигация внутри раздела (например сайдбар в dashboard)
-  разные обертки для разных групп страниц  

**Типичные ошибки**  
- забыть `{children}` в layout.tsx - страницы просто не отрендерятся
- класть в layout то, что должно меняться на каждой странице (например заголовок конкретного поста)


### Route Groups - Группировка роутов
`Route Groups` - папка в круглых скобках `(имя_группы)`, которая **не влияет на URL**, а нужна только для организации кода или разных layout для группы страниц.  

**Пример структуры**
```
app/
  (auth)/
    login/
       page.tsx   --> URL: /login
    register/
       page.tsx   --> URL: /register
  (marketing)/
    about/
       page.tsx   --> URL: /about
```
*Название папки в скобках нигде в URL не отображается - оно просто "невидимое" для роутинга*  

**Когда использовать**  
- разный layout для авторизации и основного сайта
- группировка страниц по смыслу без раздутия URL
- разделение публичной и приватной части приложения  

**Типичные ошибки**  
- путать Route Group `(name)` с динамическим роутом `[name]` - это разные вещи
- ожидать, что название группы попадет в URL - оно туда не попадает никогда


## Динамический роутинг 
*Динамический роутинг* - автоматическое создание страниц на основе структуры папок с использованием квадратных скобок [имя_переменной]
- Имя папки в скобках (например, [id]) становится именем переменной в URL
- Внутри папки обязательно должен лежать файл page.tsx  

**Пример структуры папок**  
```
app/
  post/
    [id]/
      page.tsx  --> URL: /post/1, /post/abc
```
  
**Код динамической страницы**  
```tsx
type Props = {
  params: Promise<{
    id: string;
  }>
}

const Post = async ({ params }: Props) => {
  // Вытаскиваем id из асинхронного Promise
  const { id } = await params; 

  return (
    <>
      <h1>Post Page: {id}</h1>
    </>
  )
}

export default Post
```  
*params в Next.js 15 стали асинхронными (Promise). Поэтому компонент обязательно должен быть async, а перед чтением данных ставится await.*  

**Когда использовать**  
- страница поста/товара по id
- профиль пользователя по username
- любая страница, где часть URL - это переменная  

**Типичные ошибки**
- забыть `await params` - в Next.js 15 это Promise, а не обычный объект
- забыть сделать компонент `async`, хотя используется `await`


## Серверные компоненты - По Умолчанию 

- Все файлы и компоненты внутри папки app изначально являются Серверными.  

**Плюсы**:  
- Нулевой вес для JS-бандла клиента (0 байт), моментальная загрузка, идеальное SEO, безопасное использование API-ключей прямо в коде (они не улетят в браузер).  
**Минусы**:  
- Полное отсутствие интерактива. Нельзя использовать клики, сабмиты и хуки React (useState, useEffect).  
 
**Доп**  
- Могут быть как асинхронными (`async/await` для прямого получения данных с сервера), так и обычными синхронными компонентами (если им нужно просто отрендерить статичную верстку или принять пропсы).  

**Когда использовать**  
- страницы, layout, загрузка данных
- всё, что не требует кликов/хуков/интерактива по умолчанию  

**Типичные ошибки**  
- пытаться использовать useState/useEffect в серверном компоненте - будет ошибка
- хранить в серверном компоненте секретные ключи и случайно передавать их пропсом в клиентский компонент - ключ "утечет" в браузер  


## Клиентские компоненты 
`Клиентские компоненты ` - Компоненты, которые отвечают за интерактивность и реактивность на стороне пользователя.  
  
`Как включить`: В самой первой строчке файла (до импортов) нужно написать директиву 'use client'.  

**Пример:**  
```tsx
'use client' // Переключаем компонент в клиентский режим

import { useState } from 'react'

const Counter = () => {
  const [count, setCount] = useState<number>(0)

  return (
    <button onClick={() => setCount(count + 1)}>
      Кликнули: {count}
    </button>
  )
}
export default Counter
```

**Плюсы**:      
- Разрешено использовать любые хуки React (useState, useEffect, useRef) и отслеживать любые браузерные события (onClick, onChange).   
 **Минусы**  
- Директива 'use client' не означает, что компонент рендерится только на клиенте. Его первичный HTML-каркас Next.js всё равно соберет на сервере. Поэтому вызывать браузерные объекты (типа window или localStorage) напрямую в теле компонента нельзя (будет ошибка на сервере) — их нужно прятать внутрь useEffect.  


**ГЛАВНОЕ ПРАВИЛО АРХИТЕКТУРЫ ПРИ РАБОТЕ С КОМПОНЕНТАМИ:**  
- Делать всю старницу, макеты загрузку данных `серверными`. А интерактивные элементы(кнопкиб формы) выносить в отдельные файлы, и писать там `'use client'`. и подключать и внутрь серверных страниц как обычные теги.  

**Когда использовать**  
- кнопки, формы, модалки, счетчики - всё с onClick/onChange
- использование хуков (useState, useEffect, useRef)
- работа с window, localStorage, браузерными API (внутри useEffect)  

**Типичные ошибки**  
- ставить `'use client'` на всю страницу целиком "на всякий случай" - теряется весь смысл серверных компонентов
- обращаться к `window`/`localStorage` прямо в теле компонента, а не внутри `useEffect`  


## Получение данных на сервере и кэширование

Пример
```tsx
async function getData(id:string) {
  const res = await fetch(`https://jsonplaceholder.typicode.com/posts/${id}`, {
    next:{
      revalidate: 60, 
    }, // Интеравал запросов раз в какое то время 
  })
  if (!res.ok) {
    throw new Error('Failed to fetch data');
  }
  return res.json()
}
// Пишем здесь такую же функцию которая делает запрос к серверу но только к ондному посту. получает данныеи выводит их на старницу 
```

### Варианты кэширования fetch
`fetch` в Next.js по умолчанию сам кэширует запросы. Есть 3 режима:

| Вариант | Что делает |
| --- | --- |
| `cache: 'force-cache'` | По умолчанию. Кэширует навсегда (пока не передеплоят проект) |
| `next: { revalidate: 60 }` | ISR - кэш обновляется раз в N секунд |
| `cache: 'no-store'` | Кэш выключен, данные всегда свежие (как SSR) |  

*Чем меньше revalidate - тем "живее" данные, но выше нагрузка на сервер*  

**Когда использовать**  
- `force-cache` - данные почти никогда не меняются (статьи, документация)
- `revalidate` - данные меняются, но не критично показать чуть устаревшие (каталог товаров)
- `no-store` - данные должны быть всегда актуальны (личный кабинет, баланс, чат)  

**Типичные ошибки**  
- использовать `no-store` везде "для надежности" - убивает всю выгоду от кэширования и замедляет сайт
- забыть `if (!res.ok)` - при ошибке fetch не бросает исключение сам, это нужно проверять руками  


### generateStaticParams - Статическая генерация динамических страниц
Для динамических роутов ([id]) можно заранее сгенерировать статические HTML-страницы на этапе сборки, а не рендерить их каждый раз заново на сервере (SSG вместо SSR).

```tsx
export async function generateStaticParams() {
  const posts = await fetch('https://jsonplaceholder.typicode.com/posts').then(res => res.json())

  return posts.map((post: { id: number }) => ({
    id: String(post.id),
  }))
}
// Next.js заранее соберет /post/1, /post/2 и тд ещё во время билда
```  

**Когда использовать**  
- блог
- товары
- документация
- страницы, которые редко меняются  

**Типичные ошибки**  
- использовать для страниц, у которых тысячи и тысячи id (сборка будет очень долгой) - лучше сгенерировать только популярные, а остальное дорендерить по запросу
- забыть, что `id` должен быть строкой (`String(post.id)`), а не числом - иначе не совпадет с типом params  


## Erorr и Loading 
- Создаеются два файла для отображения загрузки и ошибок. Обязательно их называть `error.tsx` , `loading.tsx` и внутрь пишется что будет возвращять страница при ошибке и загрузке. next Js автоматический их использует.  
*Erorr.tsx в обязательном порядке должен быть именно клинстким компонентом 'use client'*
```tsx
'use client';

const Error = ({error}: {error: Error}) => {
  return (
    <>
      <h1>Oops! {error.message} </h1>
    </>
  )
}

export default Error
```  

**Когда использовать**  
- `loading.tsx` - долгие запросы к API/базе, чтобы не показывать белый экран
- `error.tsx` - любая страница с fetch, который может упасть    

**Типичные ошибки**  
- забыть `'use client'` в error.tsx - обязателен, т.к. компонент использует React error boundary
- ожидать, что loading.tsx поймает ошибку - он показывается только во время загрузки, за ошибки отвечает error.tsx   


### not-found.tsx - Страница 404
Аналогично error.tsx и loading.tsx - файл `not-found.tsx` автоматически показывается, когда страница не найдена.

```tsx
const NotFound = () => {
  return (
    <>
      <h1>404 - Страница не найдена</h1>
    </>
  )
}

export default NotFound
```

- Вызвать эту страницу вручную (например если пост с таким id не найден в базе) можно функцией `notFound()` из `next/navigation`:
```tsx
import { notFound } from 'next/navigation'

async function getData(id: string) {
  const res = await fetch(`.../posts/${id}`)
  if (!res.ok) return null
  return res.json()
}

const Post = async ({ params }: Props) => {
  const { id } = await params
  const data = await getData(id)

  if (!data) {
    notFound() // Покажет содержимое not-found.tsx
  }

  return <h1>{data.title}</h1>
}
```  

**Когда использовать**  
- несуществующий id в динамическом роуте
- обычный несуществующий URL (срабатывает автоматически)  

**Типичные ошибки**  
- пытаться вернуть `notFound()` как обычное значение (`return notFound()`) - функция сама прерывает рендер, `return` перед ней не обязателен, но и не мешает


## next/link и next/image - Оптимизированные компоненты

### Link - Навигация
`Link` заменяет обычный тег `<a>`. Переход между страницами происходит без полной перезагрузки, а ссылки внутри видимой области экрана автоматически "прогреваются" (prefetch).

```tsx
import Link from 'next/link'

<Link href="/about">О нас</Link>
```

### Image - Оптимизация картинок
`Image` заменяет обычный тег `<img>`. Автоматически сжимает изображение, конвертирует в современные форматы (webp/avif) и делает ленивую загрузку (lazy loading).

```tsx
import Image from 'next/image'

<Image src="/logo.png" alt="Логотип" width={200} height={100} />
```  
*width и height обязательны для статичных импортов - так Next.js заранее знает пропорции и не "дергает" верстку при загрузке*  

**Когда использовать**  
- `Link` - вообще всегда вместо `<a>` для внутренней навигации
- `Image` - вообще всегда вместо `<img>` для картинок проекта  

**Типичные ошибки**  
- использовать обычный `<a>` для внутренних переходов - теряется prefetch и получается полная перезагрузка страницы
- не указать `width`/`height` у `Image` - будет ошибка сборки или "прыгающая" верстка  


## Metadata - SEO
Для тайтла, описания и прочих мета-тегов страницы в page.tsx или layout.tsx экспортируется объект `metadata` (или функция `generateMetadata`, если данные динамические).  

**Статичный вариант**  
```tsx
import { Metadata } from 'next'

export const metadata: Metadata = {
  title: 'Главная страница',
  description: 'Описание для поисковиков',
}
```
  
  
**Динамический вариант** (например заголовок из данных поста)  
```tsx
export async function generateMetadata({ params }: Props): Promise<Metadata> {
  const { id } = await params
  const post = await getData(id)

  return {
    title: post.title,
  }
}
```  

**Когда использовать**  
- `metadata` - статичные страницы (главная, о нас, контакты)
- `generateMetadata` - страницы с динамическим контентом (пост, товар)  

**Типичные ошибки**  
- экспортировать и `metadata`, и `generateMetadata` одновременно в одном файле - конфликт, нужно выбрать что-то одно
- забыть, что `generateMetadata` тоже асинхронная и тоже получает `params` как Promise  


## Route Handlers - Свой API 
`Route Handlers` - способ создать свой API-эндпоинт прямо внутри app-роутера. Файл обязательно называется `route.ts` и лежит в папке `app/api/...`

```
app/
  api/
    posts/
      route.ts   --> URL: /api/posts
```

```tsx
import { NextResponse } from 'next/server'

// GET /api/posts
export async function GET() {
  const posts = await fetch('https://jsonplaceholder.typicode.com/posts').then(res => res.json())
  return NextResponse.json(posts)
}

// POST /api/posts
export async function POST(request: Request) {
  const body = await request.json()
  // ... логика сохранения
  return NextResponse.json({ success: true })
}
```  
*Название экспортируемой функции = HTTP-метод (GET, POST, PUT, DELETE и тд)*  

**Когда использовать**  
- нужен эндпоинт для внешних клиентов (мобильное приложение, вебхуки)
- нужна ручка, которая не привязана напрямую к отправке формы  

**Типичные ошибки**  
- называть файл не `route.ts`, а как-то иначе - Next.js его просто не найдет
- использовать Route Handler там, где хватило бы Server Action (усложнение без необходимости)


## Server Actions - 'use server'
`Server Actions` - функции, которые выполняются строго на сервере, но их можно вызывать прямо из клиентских компонентов (например из формы), без создания отдельного API-роута.  

`Как включить`: директива `'use server'` в начале функции или файла.    

**Пример с формой**  
```tsx
// actions.ts
'use server'

export async function createPost(formData: FormData) {
  const title = formData.get('title')
  // ... сохраняем в базу
}
```

```tsx
// form.tsx
import { createPost } from './actions'

const Form = () => {
  return (
    <form action={createPost}>
      <input name="title" />
      <button type="submit">Создать</button>
    </form>
  )
}
export default Form
```  
*Плюс: не нужно вручную писать fetch и API-роут - Next.js сам "прокидывает" вызов на сервер*  

**Когда использовать**  
- отправка форм (создание поста, регистрация, комментарий)
- мутации данных (лайк, удаление, обновление), не нуждающиеся во внешнем API  

**Типичные ошибки**  
- забыть `'use server'` - функция выполнится как обычная клиентская, и обращение к базе данных упадет в браузере
- пытаться напрямую вернуть JSX или сложные объекты (классы, функции) из Server Action - можно передавать только сериализуемые данные


## Middleware
`Middleware` - код, который выполняется до того, как запрос дойдет до страницы. Используется для редиректов, проверки авторизации, работы с куками.

- Файл обязательно называется `middleware.ts` и лежит в корне проекта (рядом с app)

```tsx
import { NextResponse } from 'next/server'
import type { NextRequest } from 'next/server'

export function middleware(request: NextRequest) {
  const isAuth = request.cookies.get('token')

  if (!isAuth) {
    return NextResponse.redirect(new URL('/login', request.url))
  }

  return NextResponse.next()
}

// На какие пути распространяется middleware
export const config = {
  matcher: ['/dashboard/:path*'],
}
```  

**Когда использовать**  
- закрыть dashboard
- проверять JWT
- менять язык сайта
- делать redirect  

**Типичные ошибки**  
- забыть `matcher` - middleware по умолчанию срабатывает на всех роутах, включая статику, что бьет по производительности
- делать в middleware тяжелые запросы к базе - он выполняется на каждый запрос и должен быть максимально легким  


## redirect() - Серверный редирект
Функция `redirect()` из `next/navigation` позволяет перенаправить пользователя прямо внутри серверного компонента (например если пользователь не авторизован).

```tsx
import { redirect } from 'next/navigation'

const Dashboard = async () => {
  const isAuth = false // проверка авторизации

  if (!isAuth) {
    redirect('/login')
  }

  return <h1>Личный кабинет</h1>
}
export default Dashboard
```  

**Когда использовать**  
- редирект внутри серверного компонента после проверки условия (например нет доступа)
- редирект внутри Server Action после успешного выполнения (например после создания поста - на страницу поста)  

**Типичные ошибки**   
- использовать `redirect()` в клиентском компоненте - для этого нужен хук `useRouter()` из `next/navigation`, а не эта функция
- оборачивать `redirect()` в try/catch - он технически работает через throw, и try/catch может случайно "проглотить" редирект
